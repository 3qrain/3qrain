可以按两个阶段推进，第一阶段先把媒体库自身做好，第二阶段再引入业务引用关系。

第一阶段：媒体库（文件管理）

目标：

把媒体库做成一个基于文件系统的图床/资源管理器。
暂不关心文章、随记等业务引用关系。
存储结构
uploads/
├── posts/
├── images/
├── videos/
├── audios/
└── files/

所有实际文件都存储在文件系统中。

数据库表
media
media
--------------------------------
id
path            -- 唯一文件路径
alias           -- 文件别名（可选）

created_at
updated_at

说明：

path 唯一
不存 filename
不存 size
不存 mime_type
不存 width/height

因为这些都可以通过文件系统实时获取。

例如：

/uploads/posts/2026/06/banner.webp

文件名：

path.basename(path)

文件大小：

fs.stat(path).size

文件类型：

path.extname(path)
上传流程
上传文件
↓
保存到 uploads
↓
写入 media 表
↓
返回文件 URL

如果写库失败：

删除已上传文件

保证一致性。

删除流程
删除文件
↓
删除物理文件
↓
删除 media 记录

如果物理文件删除失败：

数据库不删除

保证一致性。

媒体库页面

展示：

文件预览
文件名称
文件路径
文件大小
文件类型
上传时间
别名

支持：

上传
删除
复制链接
预览
搜索
一致性检查

媒体库增加：

健康检查
未登记文件

文件存在：

uploads/*

数据库无记录：

media.path

这种情况属于：

未登记文件

例如：

uploads/posts/a.webp

存在

但是：

media

没有记录。

失效记录

数据库有：

media.path

记录

但是：

uploads/posts/a.webp

不存在。

属于：

失效记录
页面展示
未登记文件：3
失效记录：1

提供：

登记文件
清理记录

按钮。

第二阶段：媒体引用关系

目标：

知道一个文件被哪些内容引用。
知道一个内容引用了哪些文件。
支持孤儿文件分析。
数据库表

新增：

media_references
media_references
--------------------------------
id

media_id

target_type
target_id

created_at
target_type

第一阶段先支持：

post
note

未来可以扩展：

page
project
comment
示例

文章 12 引用了文件 5：

media_id = 5

target_type = "post"
target_id = 12

随记 8 引用了文件 5：

media_id = 5

target_type = "note"
target_id = 8
引用同步机制
创建文章

保存文章内容时：

解析内容
↓
提取所有媒体 URL
↓
匹配 media
↓
写入 media_references
更新文章
删除当前文章旧引用
↓
重新解析内容
↓
重新生成引用关系

类似：

DELETE FROM media_references
WHERE target_type='post'
AND target_id=?

然后重新插入。

删除文章
删除文章
↓
删除对应引用关系

不删除文件。

媒体库增强

每个文件增加：

引用次数
引用来源

例如：

banner.webp

被引用：
- 文章《Vue3 实战》
- 文章《Tiptap 入门》
- 随记《测试内容》
孤儿文件分析

查询：

存在于 media
但不存在于 media_references

即可得到：

孤儿文件

例如：

abc.webp

上传过。

数据库有记录。

但没有任何文章、随记引用。

可以：

删除
批量删除

这样设计层次清晰，而且后续从第一阶段升级到第二阶段时，数据库结构和媒体库页面都不用推翻重做。
